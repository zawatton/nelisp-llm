#+title: nelisp-llm

* Overview

~nelisp-llm~ is the *experiment repository* for building and training small
language models in Emacs Lisp / NeLisp.  It is deliberately separate from
~nelisp-photon~:

- ~nelisp-photon~ = the stable *library*: an implementation of Fujitsu's
  PHOTON architecture plus the tensor core (~photon-tensor~), the GPT-style
  dense baseline (~photon-transformer~), tokenizer (~photon-bpe~), weight
  IO (~photon-weights~), the hardware autosizer (~photon-spec~), and the
  GPU backend (sibling ~nelisp-gpu~).  API is kept clean and NeLisp-loadable.
- ~nelisp-llm~ (this repo) = the *lab*: depends on ~nelisp-photon~ and moves
  fast, adopting modern open-weight techniques and training on fully-open
  data.  Breaking changes are fine here.

* Direction

Adopt, on top of the ~nelisp-photon~ substrate, the architectural techniques
shared by the strongest open-weight models (GLM, DeepSeek, Qwen) that the
plain GPT baseline lacks:

- *RMSNorm* (instead of LayerNorm)        -- implemented, trainable
- *RoPE* rotary position embeddings        -- implemented, trainable
- *SwiGLU* / SiLU feed-forward             -- implemented, trainable
- *KV cache* (incremental decode)          -- implemented (decode == full)
- *GQA* grouped-query attention            -- implemented, trainable
- *MoE* mixture-of-experts                  -- implemented, trainable

Every primitive has both a forward path and a reverse-mode autograd backward
(numerically gradient-checked), so the full modern block -- multi-head GQA with
RoPE plus a top-k sparse MoE FFN -- is *directly trainable*, not only run
forward.  See ~examples/train-modern-full.el~.

Training data: fully-open corpora only -- The Pile (EleutherAI), FineWeb /
FineWeb-Edu, or Dolma -- so the whole pipeline (data + code + weights) stays
reproducible.

* GPU acceleration

The heavy linear algebra runs on the GPU via the sibling ~nelisp-gpu~
(SPIR-V/Vulkan) backend, reusing ~photon-tensor-gpu~'s function-cell swap, so
the modern block runs on the GPU unchanged.  ~nl-llm-gpu-enable~ starts the
persistent vkserver and routes ~photon-tensor~ matmul/linear/softmax through
the GPU kernels; both the forward projections / expert FFNs and the attention
score/context matmuls (in the autograd path) are accelerated.

Weights are kept *resident* on the GPU (uploaded once, referenced by handle).
For training, the in-place SGD step mutates weights the resident cache cannot
see, so ~nl-llm-gpu-invalidate~ must be called after each step to refresh the
GPU copies; ~test/gpu-test.el~ verifies this end-to-end (and that the GPU
forward matches the CPU result, and that the GQA+MoE block actually trains on
the GPU).

Measured on a GeForce GTX 1060 (one block, seq 16, ff = 4·dim, ~make bench-gpu~):

| size     | op      | CPU ms | GPU ms | speedup |
|----------+---------+--------+--------+---------|
| dim 64   | infer   |    367 |     73 |   5.0x  |
| dim 128  | infer   |   1213 |    131 |   9.3x  |
| dim 256  | infer   |   4460 |    261 |  17.1x  |
| dim 64   | train   |   1224 |   1360 |   0.9x  |
| dim 128  | train   |   4125 |   4234 |   1.0x  |
| dim 256  | train   |  15329 |  14551 |   1.1x  |

Inference is a large, *scaling* win (residency amortises the weight upload over
every token).  Host-driven training is break-even: in-place SGD forces the
(mutated) weights to be re-encoded and re-uploaded every step, and the elisp
float marshalling eats the GPU compute gain.

** On-device training (resident weights + on-device SGD)

The fix is to keep the weights *and* the optimiser update on the GPU.
~nl-llm-gpu-mlp-train~ (in ~nl-llm-gpu-train.el~) uploads the weights, inputs
and targets to resident buffers once, then runs each step's *entire* forward +
backward + SGD as ONE fused command buffer that updates the resident weight
buffers in place (a new ~sgd~ GPU kernel writes back into the resident slots,
alongside new ~sub-scale~ / ~colsum~ / ~gelu-bwd~ kernels).  Nothing
weight-sized crosses the host boundary per step.  ~test/gpu-train-test.el~
checks the on-device loss trajectory matches the CPU autograd reference
(maxrel < 2e-2).

Per-step training time for an FFN-shaped MLP (dim -> 4·dim -> dim, seq 16,
steady state via differencing; ~make bench-gpu-train~):

| MLP            | CPU ms | host-GPU ms | on-device ms | vs CPU | vs host-GPU |
|----------------+--------+-------------+--------------+--------+-------------|
| 64 ->256 ->64  |    505 |         606 |          5.4 |   93x  |   112x      |
| 128->512 ->128 |   1905 |        2059 |          9.7 |  197x  |   213x      |
| 256->1024->256 |   7413 |        7348 |         22.8 |  325x  |   322x      |

Keeping the parameters resident and updating them on-device turns training from
break-even into a 90-320x win that *grows* with model size, because the per-step
cost is now GPU compute + a tiny command, not elisp weight marshalling.

*** Resident-autograd builder: the full block on-device

To go beyond the hand-wired MLP, ~nl-llm-gpu-ag.el~ is a small deferred autograd
whose tensors live on the GPU: each op emitter appends its forward dispatch and
records a backward thunk; running the thunks in reverse emits the backward
dispatches (accumulating gradients into resident grad slots) and a final pass
emits an on-device ~sgd~ per parameter -- the whole forward + backward + SGD is
ONE fused batch per step with the weights resident.  A model is expressed once
(~nlga-linear~ / ~rmsnorm~ / ~rope~ / ~gqa~ / ~swiglu~ / ~softmax~ / ...) and is
trained on-device automatically.

The full modern block -- RMSNorm + grouped-query attention with RoPE + a feed
-forward (SwiGLU, or a top-k sparse MoE) + an LM head with cross-entropy --
trains entirely on-device this way.  ~test/gpu-block-test.el~ (dense/SwiGLU) and
~test/gpu-moe-test.el~ (GQA + MoE) check the loss trajectory matches the CPU
autograd reference (~nl-llm-ag-block~) to f32 precision (maxrel < 1e-7).  For
MoE the top-k routing selection is computed on-GPU (~topk-mask~) and held
constant (straight-through); the gate softmax and expert SwiGLUs are
differentiated.  New backward kernels added for the block:
~rmsnorm-istd/-fwd/-dx/-dgamma~, ~rope-apply~, ~silu~ / ~silu-bwd~, ~mul~,
~softmax-bwd~, ~ce-grad~, and for MoE ~topk-mask~ / ~scale-rows~ / ~rowdot~.

A whole stacked model trains on-device too: ~nlga-model~ embeds (a one-hot @ a
resident ~wte~, so the embedding is trained by the matmul backward), stacks N
blocks, then RMSNorm + head.  ~test/gpu-stack-test.el~ checks a 2-block model
(dense block + MoE block) with a trainable embedding matches the CPU autograd
reference to f32 (maxrel < 1e-7); ~examples/train-stacked-gpu.el~ trains a
2-block dim-64 model on public-domain text (loss 5.75 -> 0.04, forward +
backward + SGD all resident).

*** Compiled batches (record once, re-submit)

During a training loop the graph (slots + dispatches) is *identical* every step
-- only the resident weight buffers change in place.  ~nlga-compile~ records the
whole fused command buffer once on the server (~OP_COMPILE~): it allocates the
tmp/grad buffers once, binds the resident weights, records every dispatch with a
GPU-side ~vkCmdFillBuffer~ zeroing of the tmp/grad buffers up front, and returns
a handle.  Each subsequent ~nlga-step~ just re-submits that command buffer
(~OP_RUN_COMPILED~) -- no per-step protocol re-send, buffer allocation, or
descriptor rebuild.  This is bit-identical to the per-step batch path (max loss
diff 0) and ~10x faster: on the 2-block dim-64 model the GPU per-step time drops
from ~108 ms to ~11 ms on a GTX 1060.

*** Training over many windows (corpus training)

A compiled batch can be re-run on fresh data without recompiling: the one-hot
token/target inputs are resident buffers, and ~nlga-update~ overwrites their
contents in place (~OP_WRITE_RESIDENT~).  So a real training loop compiles the
fused batch once and, per window, just refreshes the (small) one-hot inputs and
re-submits -- every weight stays resident, nothing weight-sized moves.
~test/gpu-window-test.el~ checks a sliding-window run matches CPU autograd to f32
(maxrel < 1e-7); ~examples/train-corpus-gpu.el~ trains a 2-block dim-64 model
over thousands of windows of public-domain text on-device (avg loss 5.6 -> 4.7).

The embedding is a *gather* (~nlga-embed~): the per-window input is just seq
token + seq target indices (the ~embed-gather~ / ~embed-bwd~ / ~ce-grad-idx~
kernels index ~wte~ directly via a float->uint cast), not a seq x vocab one-hot.
~test/gpu-gather-test.el~ confirms it matches CPU autograd
(~photon-autograd-embedding~) to f32, and it cuts the corpus per-window time from
~38 ms to ~20 ms (vocab 336).  ~wte~ is still trained on-device by the
scatter-add backward.

*** On-device Adam

~nlga-finish-adam~ swaps the plain SGD update for Adam, fully on-device: each
parameter's first/second moment buffers (m, v) are kept resident and updated in
place by the ~adam~ kernel (like the weights, they never round-trip), and a
shared resident buffer holds [lr_t, b1, b2, eps].  Because the compiled command
buffer is fixed, the time-dependent bias correction is handled by refreshing
lr_t = lr·sqrt(1-b2^t)/(1-b1^t) in that buffer each step via
~nlga-adam-update-t~.  ~test/gpu-adam-test.el~ checks it matches a CPU reference
running the same Adam update to f32 (maxrel < 1e-5); Adam reaches a much lower
loss in the same number of steps than SGD (2.89 -> 0.64 in 8 steps here).

*** Fused multi-head attention

The attention core no longer loops over heads in elisp emitting O(heads)
dispatches.  ~nlga-gqa~ now computes all heads in a single dispatch per stage:
~attn-scores~ (Q·K^T with the 1/sqrt(hd) scale and causal mask folded in, output
laid out (heads*seq) x seq so the existing row-softmax applies directly),
~attn-context~ (the P·V context), and matching batched backward kernels
(~attn-sc-dq/dk~, ~attn-ctx-dp/dv~) -- so the per-block attention dispatch count
is constant in the number of heads instead of growing with it.  Bit-for-bit the
same math (the block/MoE/stack tests still match CPU autograd to f32).

*** Fused MLP / SwiGLU

Two more fusions cut the feed-forward dispatch count: ~matmul-at~ computes a
linear's weight gradient dW = g^T·x in a single dispatch (no separate transpose
op or buffer), shrinking *every* linear's backward; and ~silu-mul~ /
~silu-mul-bwd~ fold the SwiGLU gate silu(a)·b into one forward and one backward
kernel.  Same math (still f32-identical to CPU); on the 2-block dim-64 model the
batch shrinks from 257 to 234 dispatches.

A weight's gradient has no fan-out (each weight is used once), so its backward
op writes straight into the resident grad slot (zeroed each run) instead of
computing into a tmp and vadd-accumulating -- a tmp + a vadd saved per weight,
bias, RMSNorm gain and embedding table.  (A ~gwritten~ flag keeps it correct if a
parameter ever is reused: the first write is direct, later ones accumulate.)
Activation gradients, which do fan out, keep the accumulate path.  The 2-block
batch drops further to 198 dispatches; per-step ~17 ms.

*** Weight tying (head = wte^T)

Because ~nlga-linear~ computes x·W^T, the LM head can share the embedding matrix
simply by passing the embedding ~wte~ as the head's weight -- no separate output
matrix.  ~wte~ is then used twice (gather embedding + head), so its gradient is
accumulated from both backward passes; the ~gwritten~ fan-out path handles this
exactly (the first contribution writes the grad, the second vadds).
~test/gpu-tie-test.el~ trains a tied model on-device and matches the CPU autograd
reference (which ties the same ~wte~ pav) to f32 (maxrel ~9e-8).

*** Learning-rate schedules

The learning rate is just a value in a resident buffer, so a schedule needs no
new machinery: for SGD, ~nlga-update~ the lr scalar each step; for Adam, pass the
scheduled base lr to ~nlga-adam-update-t~ (it still applies the bias correction).
~nl-llm-lr-warmup-cosine~ and ~nl-llm-lr-inv-sqrt~ provide the usual shapes.
~test/gpu-sched-test.el~ trains with a warmup+cosine schedule and matches a CPU
reference using the same per-step lr to f32 (maxrel ~8e-6).

*** Regularisation: gradient clipping and dropout

Global-norm gradient clipping is on-device: after backward, ~sumsq-acc~ reduces
every parameter gradient into one sum-of-squares slot, ~clip-scale~ computes
scale = min(1, clip/||g||), and the SGD/Adam kernels multiply the gradient by it
\(pass ~clip~ to ~nlga-finish~ / ~nlga-finish-adam~).  ~test/gpu-clip-test.el~
matches a CPU global-norm clip to f32 and checks the clip actually engages.

Dropout is inverted-dropout by a refreshable mask: ~nl-llm-dropout-mask~ builds a
mask (each element 1/(1-p) with prob 1-p else 0, mean 1), and ~nlga-dropout~
elementwise-multiplies it (gradient passes through the mask).  Refresh the mask
per step with ~nlga-update~; ~test/dropout-test.el~ checks the mask statistics.

*** Checkpoints (save / resume)

~nl-llm-ckpt-save~ / ~nl-llm-ckpt-load~ persist a trained model -- config, step,
and every tensor (embedding, blocks with biases, final gain, tied-head bias) --
to a sexp file (photon-tensors are plain [shape data] vectors and floats
round-trip exactly through prin1/read).  ~test/ckpt-test.el~ checks the round-trip
is exact and a bad format tag is rejected; ~examples/checkpoint-gpu.el~ trains a
model, saves it, STOPS the GPU, then reloads from disk, restarts the GPU and
generates from the reloaded weights -- reproducing the passage 24/24, i.e. the
model survives a full session-end / reload cycle.

The checkpoint also carries the Adam optimiser state (~nlga-adam-state~ reads the
resident m/v back, ~nlga-adam-restore~ writes them back) and the step, so resume
is *seamless*: ~test/gpu-resume-test.el~ trains 2K steps straight and, separately,
trains K, checkpoints weights+optimiser+step, reloads, restores and trains K more
-- the resumed loss trajectory matches the continuous one bit-for-bit (maxdiff 0),
because every resident buffer round-trips exactly through f32.

*** Full recipe in one run

~examples/train-full-gpu.el~ runs the whole recipe together: train a weight-tied
stacked model over sliding windows of the corpus, on-device, with Adam +
warmup/cosine LR + global-norm gradient clipping + dropout (mask refreshed each
step), then generate via the KV-cache decode with both greedy and
high-temperature top-k sampling.  On the tiny demo model the corpus loss falls
(avg 5.5 -> 4.5); greedy collapses to the most frequent token (expected for a
small model), while temperature sampling stays varied and non-degenerate:

: prompt        : "In another momen"
: greedy        : "In another momen eo o o o o o ..."
: sampled(T1.0) : "In another momenb, eabr ouse eor, ok she e..."

*** End-to-end: train on-device, then generate

~examples/generate-gpu.el~ closes the loop: BPE-encode a passage, train a
weight-tied stacked model on it fully on-device (gather embedding, Adam,
warmup+cosine LR -- every weight resident), read the trained weights back, then
autoregressively generate from a prompt and BPE-decode.  Generation is
forward-only and reuses the CPU autograd forward (the same model the on-device
path trained).  On a tiny model (2 blocks, dim 64) that memorises one window, it
reproduces the learned text exactly:

: prompt        : "In another momen"
: greedy        : "In another moment down went Alice after it, never"
: sampled(T0.7) : "In another moment down went Alice after it, never"
: greedy match  : 24/24  reproduced

Decoding supports both greedy (argmax) and temperature / top-k sampling
(~nl-llm-sample~: temperature scale -> top-k mask -> softmax -> categorical draw;
T->0 or top-k=1 reduce to greedy).  ~test/sample-test.el~ checks the sampler
deterministically (seeded RNG): top-k restricts the support, temperature->0 is
argmax, a peaked logit is the mode, a uniform one spreads.

Decoding uses a KV cache (~nl-llm-decode~): each block keeps a key/value cache
and only the new token's projections + attention over the cache are computed per
step, so generating token t costs ~O(dim^2 + t·hd) instead of re-running the
whole O(t^2) prefill every step.  The cache path supports biases and the tied
head and is verified position-for-position against the prefill forward
(~test/decode-test.el~, maxdiff ~1e-15).

Decoding can also run entirely on the GPU (~nl-llm-gpu-decode~): the weights and a
per-block key/value cache stay resident, and each token is one fused command
buffer (compiled once, re-submitted) computing only the new position --
projections + RoPE at a runtime position, an append into the resident cache, and
a single-query attention over the cache whose length is read from a resident POS
buffer (the attention kernel's loop bound is a runtime value).
~test/gpu-decode-test.el~ matches the CPU decode to f32 (maxrel ~3e-5);
~examples/generate-gpu.el~ uses it.

Batched decode (~nl-llm-gpu-decode-batch~) generates B sequences in parallel:
each sequence has its own resident key/value cache, and one fused batch processes
B new tokens per step (B-row projections + ~decode-rope-b~ / ~cache-append-b~ /
~decode-attn-b~ kernels, shared position).  ~test/gpu-batch-test.el~ checks each
sequence's logits match decoding it alone (maxrel ~8e-6).

*** Bounded-memory decode: StreamingLLM attention sinks

~nl-llm-stream.el~ bounds the KV cache for arbitrarily long generation (Xiao et
al. 2023).  A pure sliding window collapses once the first tokens are evicted,
because softmax dumps its surplus probability mass onto the first few tokens --
the /attention sinks/.  Keeping =nsink= initial tokens permanently plus a rolling
window of the last =win= tokens bounds the cache at =nsink+win= entries while
preserving quality.  The one subtlety is RoPE: cached keys are rotated by their
/cache-relative/ position (rank in the kept set), not their absolute stream
position, so every relative distance the query sees stays in distribution.  We
store raw keys and rotate at attention time.

When the cap is never reached this reduces *exactly* to ~nl-llm-decode~:
~test/stream-test.el~ checks (a) cap>=len gives bit-identical logits to the plain
KV decode (maxdiff 0), (b) the cache stays bounded with sink slots pinned, and
(c) under period-=win= input the logits repeat with period =win= (bit-identical),
pinning the cache-relative RoPE + ring indexing.  ~examples/stream-decode.el~
trains a model and compares full vs sink+window vs window-only decode at an equal
cache budget (the sink benefit is a learned effect, hence the training step).

The same scheme runs on the GPU (~nl-llm-gpu-stream-new~ / ~-step~): keys/values
are stored RAW in a cap=(nsink+win) ring cache, and one fused kernel
(~decode-attn-stream~) rotates the query by its cache-relative position and each
key by its cache-relative rank at attention time, deriving the ring slot, window
offset and cache-relative positions in-kernel from the resident POS -- so the
command buffer is compiled once and the cache stays bounded forever.
~cache-append-ring~ writes raw projections at the ring slot.
~test/gpu-stream-test.el~ matches the on-GPU decode to the CPU oracle to f32
(maxrel ~1e-5 under eviction; ~4e-5 vs full decode when the cap is never reached).

*** Ternary weights: BitNet b1.58 (QAT, Phase A)

~nlga-bitlinear~ trains a linear with *ternary* weights and *int8* activations
(Microsoft BitNet b1.58): the effective weight is =beta * round(clip(W/beta, -1,
1))= with =beta = mean|W|=, so it takes values in {-beta, 0, +beta} -- the "1.58
bits" -- and each activation row is absmax-quantized to int8.  Training keeps a
full-precision latent W; ~nlga-quant-weight~ (kernels =absmean-acc= + =quant-w=)
and ~nlga-quant-act~ (kernel =quant-act=) quantize on-device each step, and both
backwards are straight-through estimators (the quantizer is identity in reverse,
so the latent W gets the gradient and the optimizer trains it).
~test/gpu-bitnet-test.el~ checks the GPU BitLinear against a CPU STE reference
with both quantizers: forward matches to maxrel 1.9e-7, the training loss
trajectory matches to 1.5e-7 and decreases, and the trained latent weight
quantizes to a genuinely ternary set.

Phase B packs the trained ternary weight for compact storage:
~nl-llm-bitnet-pack~ encodes it as base-4 codes, 8 per f32 (16 of 32 bits), and
the ~bitlinear-packed~ kernel reads each packed float once and peels the codes
with integer =%4= / =/4= -- so the weight buffer is ~8x smaller (the GTX 1060's
bandwidth/VRAM is the binding constraint).  ~test/gpu-bitpack-test.el~ shows the
packed forward equals the f32 ternary linear (maxrel 2.1e-7) at 8x less weight
memory.  ~examples/bitnet-model.el~ wires packed weights through a whole-model
decode: on a dim-128 / 6-block model the packed decode is bit-identical to the
f32 ternary decode and the block-linear weight VRAM drops 8.0x (5760 -> 720 KB),
6.9x for the whole model.

For *compute* (not just memory) the GTX 1060's hardware DP4A is used: the SPIR-V
compiler emits =OpSDot= (signed int8x4 dot, =VK_KHR_shader_integer_dot_product=),
four int8 lanes carried in two f32 halves (=lo + hi*65536=, no new dtype).  Kernel
~bitlinear-dp4a~ (int8 activations + ternary weights) replaces =in= f32 MACs/row
with =in/4= hardware =OpSDot=.  ~test/gpu-dp4a-test.el~ verifies it (OpSDot exact,
matmul maxrel 9e-8) and ~examples/bench-dp4a.el~ measures *1.84x vs f32 dense,
1.73x vs f32 ternary* on a resident-weight 1024x1024 matmul.

*** Paged KV cache: PagedAttention

~nl-llm-gpu-paged-new~ / ~-step~ decode B sequences whose KV lives in a shared
per-layer *pool* of fixed-size blocks (vLLM PagedAttention).  A layer-independent
block *table* (B x blocks-per-seq) maps each sequence's logical block to a
physical block id, and a host allocator hands out blocks *on demand* as positions
cross block boundaries -- so memory is reserved per used block, not B x max_seq up
front.  The =cache-append-paged= and =decode-attn-paged= kernels read/write the
pool through the table (a data-dependent dynamic index, first de-risked by
~test/gpu-paged-spike-test.el~ -- gather + scatter through a permuted table, exact).
The physical layout is interleaved (=phys = logical*B + seq=), so the table
indirection is exercised for real.  ~test/gpu-paged-test.el~ matches the CPU
per-sequence decode to f32 (maxrel 8.1e-6) and checks the on-demand block count.

The variable-length path (~nl-llm-gpu-paged-v-new~ / ~-step~ + a
~nl-llm-paged-alloc~) gives each sequence its own position (a =LENS[s]= buffer
read by ~decode-rope-b-v~ / ~cache-append-paged-v~ / ~decode-attn-paged-v~), a
host *free-list* that recycles a finished sequence's blocks, and whole-block
*prefix sharing* (several sequences point their prompt blocks at one physical
copy).  ~test/gpu-paged-v-test.el~: a batch of differing-length sequences matches
per-sequence CPU decode (maxrel 1.8e-6), freed blocks are reused, and a shared
prefix adds no blocks while the sharer's continuation is bit-identical to the
owner's (~docs/design/03-paged-attention.org~).

*** Faster decode: self-speculative (MTP) generation

~nl-llm-spec.el~ decodes with speculative execution (Leviathan et al. 2023) using
an MTP look-ahead head (Medusa/Cai et al. 2024).  Training adds a second head H2
that predicts the token /two/ ahead (a second cross-entropy at a +2 target); at
decode each round drafts the next token from the main head and a speculative token
from H2, verifies both in one target forward, and emits two tokens on a hit, one
on a miss.  The emitted stream is *exactly* plain greedy regardless of H2 -- the
acceptance only ever keeps =argmax(main head)= tokens -- so it is lossless by
construction.  ~test/spec-test.el~ pins this on a random model across seeds (no
GPU); ~examples/spec-decode.el~ trains H2 on-device and measures the speedup.

Beyond greedy, ~nl-llm-spec-sample-decode~ does *lossless sampling*: the
rejection rule (~nl-llm-spec-rejection~, accept w.p. min(1,p/q) else residual)
makes the emitted token distributed *exactly* as plain temperature/top-k sampling
regardless of the draft -- ~test/spec-test.el~ pins this statistically (output
histogram ~ target P, max|hist-P| 0.005 over 40k draws).  ~nl-llm-spec-greedy-tree~
adds a width-K *tree draft* (accept the look-ahead token when the true token is in
head2's top-K) -- still bit-exact plain greedy, higher K -> more tokens/forward.
On a trained H2: single-draft 1.92x, tree(k=4) 2.00x, lossless sampling
(T0.8/top-k30) 1.78x tokens/forward.  The core of a GPU-batched tree verify is in
place too: the ~tree-attn~ kernel computes every draft-tree node's attention over
a shared context cache + its ancestor chain in ONE dispatch
(~test/gpu-tree-attn-test.el~, maxrel 3.9e-6), and ~nl-llm-gpu-tree-verify~ wires
it through the whole decode loop -- accepted context prefill, then all M tree nodes
through every block (per-node Q/K/V + RoPE + tree-attn + SwiGLU + head) in ONE
fused forward, yielding every node's next-token logits.  ~test/gpu-tree-verify-test.el~
matches decoding each node's full path on the CPU (depth-2 tree, maxrel 7.8e-6).
With several MTP heads (head2..head4), ~nl-llm-gpu-spec-chain-decode~ drafts a
depth-D chain and verifies it in one tree-verify forward per round, accepting the
longest greedy match: ~examples/spec-chain.el~ (trained) shows tokens/forward
rising with depth -- 2.86 (D=2), 4.00 (D=3), 5.00 (D=4), all lossless.  The same
verify drives lossless *sampling* (~nl-llm-gpu-spec-chain-sample-decode~): the
rejection rule over the chain makes the emitted stream distributed exactly as
plain temperature/top-k sampling (~docs/design/04-speculative-decoding.org~).

*** Capstone: a real(er) run with mid-run checkpoint + resume

~examples/train-big-gpu.el~ exercises the whole stack at once at a larger scale: a
weight-tied 4-block model (dim 128, GQA heads 8 / kv 4, ff 512, seq 48) is trained
on-device over corpus sliding windows with Adam + warmup/cosine LR + global-norm
gradient clipping + dropout.  Halfway through (step 400) it checkpoints, the GPU
backend is *stopped* (simulating the end of a session), then the checkpoint is
reloaded -- weights, Adam state (m/v) and step counter -- the GPU is restarted, and
training *resumes*.  The loss continues across the boundary without regression
(phase 1 ends ~5.10, phase 2 picks up at ~5.13 and keeps falling to ~5.00 by step
800), confirming the resume is seamless in a real run, not just bit-identical in a
unit test.  It finishes by batch-generating from four prompts with temperature /
top-k sampling and decoding them.  ~800 steps in ~377s (~0.47s/step) on a GTX 1060.

*** Scaling: on-device vs host-driven training

Per training step of one full block (RMSNorm + GQA/RoPE + SwiGLU + head, GTX
1060, ~make bench-ondevice~), comparing the on-device stack (compiled batch,
fused attention, all weights resident, on-device SGD) against the host-driven
paths (photon-autograd on the CPU, and on the GPU backend with a per-step weight
re-upload):

| model          | CPU ms | host-GPU ms | on-device ms | vs CPU | vs host-GPU |
|----------------+--------+-------------+--------------+--------+-------------|
| dim 128 seq 32 |   7905 |        5057 |         44.4 |  178x  |   114x      |
| dim 256 seq 32 |  29041 |       15966 |         73.1 |  397x  |   218x      |
| dim 256 seq 64 |  57933 |       19590 |         86.5 |  670x  |   226x      |

The on-device advantage *grows* with model size (178x -> 670x vs CPU): the host
paths are dominated by elisp marshalling that scales with the weight/activation
size, while on-device keeps everything resident and only moves the loss out.
