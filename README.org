#+title: nelisp-llm

* Overview

~nelisp-llm~ is the *experiment repository* for building and training small
language models in Emacs Lisp / NeLisp.  It is deliberately separate from
~nelisp-photon~:

- ~nelisp-photon~ = the stable *library*: an implementation of Fujitsu's
  PHOTON architecture plus the tensor core (~photon-tensor~), the GPT-style
  dense baseline (~photon-transformer~), tokenizer (~photon-bpe~), weight
  IO (~photon-weights~), the hardware autosizer (~photon-spec~), and the
  GPU backend (sibling ~nelisp-gpu~).  API is kept clean and NeLisp-loadable.
- ~nelisp-llm~ (this repo) = the *lab*: depends on ~nelisp-photon~ and moves
  fast, adopting modern open-weight techniques and training on fully-open
  data.  Breaking changes are fine here.

* Direction

Adopt, on top of the ~nelisp-photon~ substrate, the architectural techniques
shared by the strongest open-weight models (GLM, DeepSeek, Qwen) that the
plain GPT baseline lacks:

- *RMSNorm* (instead of LayerNorm)        -- implemented, trainable
- *RoPE* rotary position embeddings        -- implemented, trainable
- *SwiGLU* / SiLU feed-forward             -- implemented, trainable
- *KV cache* (incremental decode)          -- implemented (decode == full)
- *GQA* grouped-query attention            -- implemented, trainable
- *MoE* mixture-of-experts                  -- implemented, trainable

Every primitive has both a forward path and a reverse-mode autograd backward
(numerically gradient-checked), so the full modern block -- multi-head GQA with
RoPE plus a top-k sparse MoE FFN -- is *directly trainable*, not only run
forward.  See ~examples/train-modern-full.el~.

Training data: fully-open corpora only -- The Pile (EleutherAI), FineWeb /
FineWeb-Edu, or Dolma -- so the whole pipeline (data + code + weights) stays
reproducible.

* GPU acceleration

The heavy linear algebra runs on the GPU via the sibling ~nelisp-gpu~
(SPIR-V/Vulkan) backend, reusing ~photon-tensor-gpu~'s function-cell swap, so
the modern block runs on the GPU unchanged.  ~nl-llm-gpu-enable~ starts the
persistent vkserver and routes ~photon-tensor~ matmul/linear/softmax through
the GPU kernels; both the forward projections / expert FFNs and the attention
score/context matmuls (in the autograd path) are accelerated.

Weights are kept *resident* on the GPU (uploaded once, referenced by handle).
For training, the in-place SGD step mutates weights the resident cache cannot
see, so ~nl-llm-gpu-invalidate~ must be called after each step to refresh the
GPU copies; ~test/gpu-test.el~ verifies this end-to-end (and that the GPU
forward matches the CPU result, and that the GQA+MoE block actually trains on
the GPU).

Measured on a GeForce GTX 1060 (one block, seq 16, ff = 4·dim, ~make bench-gpu~):

| size     | op      | CPU ms | GPU ms | speedup |
|----------+---------+--------+--------+---------|
| dim 64   | infer   |    367 |     73 |   5.0x  |
| dim 128  | infer   |   1213 |    131 |   9.3x  |
| dim 256  | infer   |   4460 |    261 |  17.1x  |
| dim 64   | train   |   1224 |   1360 |   0.9x  |
| dim 128  | train   |   4125 |   4234 |   1.0x  |
| dim 256  | train   |  15329 |  14551 |   1.1x  |

Inference is a large, *scaling* win (residency amortises the weight upload over
every token).  Host-driven training is break-even: in-place SGD forces the
(mutated) weights to be re-encoded and re-uploaded every step, and the elisp
float marshalling eats the GPU compute gain.

** On-device training (resident weights + on-device SGD)

The fix is to keep the weights *and* the optimiser update on the GPU.
~nl-llm-gpu-mlp-train~ (in ~nl-llm-gpu-train.el~) uploads the weights, inputs
and targets to resident buffers once, then runs each step's *entire* forward +
backward + SGD as ONE fused command buffer that updates the resident weight
buffers in place (a new ~sgd~ GPU kernel writes back into the resident slots,
alongside new ~sub-scale~ / ~colsum~ / ~gelu-bwd~ kernels).  Nothing
weight-sized crosses the host boundary per step.  ~test/gpu-train-test.el~
checks the on-device loss trajectory matches the CPU autograd reference
(maxrel < 2e-2).

Per-step training time for an FFN-shaped MLP (dim -> 4·dim -> dim, seq 16,
steady state via differencing; ~make bench-gpu-train~):

| MLP            | CPU ms | host-GPU ms | on-device ms | vs CPU | vs host-GPU |
|----------------+--------+-------------+--------------+--------+-------------|
| 64 ->256 ->64  |    505 |         606 |          5.4 |   93x  |   112x      |
| 128->512 ->128 |   1905 |        2059 |          9.7 |  197x  |   213x      |
| 256->1024->256 |   7413 |        7348 |         22.8 |  325x  |   322x      |

Keeping the parameters resident and updating them on-device turns training from
break-even into a 90-320x win that *grows* with model size, because the per-step
cost is now GPU compute + a tiny command, not elisp weight marshalling.

*** Resident-autograd builder: the full block on-device

To go beyond the hand-wired MLP, ~nl-llm-gpu-ag.el~ is a small deferred autograd
whose tensors live on the GPU: each op emitter appends its forward dispatch and
records a backward thunk; running the thunks in reverse emits the backward
dispatches (accumulating gradients into resident grad slots) and a final pass
emits an on-device ~sgd~ per parameter -- the whole forward + backward + SGD is
ONE fused batch per step with the weights resident.  A model is expressed once
(~nlga-linear~ / ~rmsnorm~ / ~rope~ / ~gqa~ / ~swiglu~ / ~softmax~ / ...) and is
trained on-device automatically.

The full modern block -- RMSNorm + grouped-query attention with RoPE + a feed
-forward (SwiGLU, or a top-k sparse MoE) + an LM head with cross-entropy --
trains entirely on-device this way.  ~test/gpu-block-test.el~ (dense/SwiGLU) and
~test/gpu-moe-test.el~ (GQA + MoE) check the loss trajectory matches the CPU
autograd reference (~nl-llm-ag-block~) to f32 precision (maxrel < 1e-7).  For
MoE the top-k routing selection is computed on-GPU (~topk-mask~) and held
constant (straight-through); the gate softmax and expert SwiGLUs are
differentiated.  New backward kernels added for the block:
~rmsnorm-istd/-fwd/-dx/-dgamma~, ~rope-apply~, ~silu~ / ~silu-bwd~, ~mul~,
~softmax-bwd~, ~ce-grad~, and for MoE ~topk-mask~ / ~scale-rows~ / ~rowdot~.

A whole stacked model trains on-device too: ~nlga-model~ embeds (a one-hot @ a
resident ~wte~, so the embedding is trained by the matmul backward), stacks N
blocks, then RMSNorm + head.  ~test/gpu-stack-test.el~ checks a 2-block model
(dense block + MoE block) with a trainable embedding matches the CPU autograd
reference to f32 (maxrel < 1e-7); ~examples/train-stacked-gpu.el~ trains a
2-block dim-64 model on public-domain text (loss 5.75 -> 0.04, forward +
backward + SGD all resident).

*** Compiled batches (record once, re-submit)

During a training loop the graph (slots + dispatches) is *identical* every step
-- only the resident weight buffers change in place.  ~nlga-compile~ records the
whole fused command buffer once on the server (~OP_COMPILE~): it allocates the
tmp/grad buffers once, binds the resident weights, records every dispatch with a
GPU-side ~vkCmdFillBuffer~ zeroing of the tmp/grad buffers up front, and returns
a handle.  Each subsequent ~nlga-step~ just re-submits that command buffer
(~OP_RUN_COMPILED~) -- no per-step protocol re-send, buffer allocation, or
descriptor rebuild.  This is bit-identical to the per-step batch path (max loss
diff 0) and ~10x faster: on the 2-block dim-64 model the GPU per-step time drops
from ~108 ms to ~11 ms on a GTX 1060.

*** Training over many windows (corpus training)

A compiled batch can be re-run on fresh data without recompiling: the one-hot
token/target inputs are resident buffers, and ~nlga-update~ overwrites their
contents in place (~OP_WRITE_RESIDENT~).  So a real training loop compiles the
fused batch once and, per window, just refreshes the (small) one-hot inputs and
re-submits -- every weight stays resident, nothing weight-sized moves.
~test/gpu-window-test.el~ checks a sliding-window run matches CPU autograd to f32
(maxrel < 1e-7); ~examples/train-corpus-gpu.el~ trains a 2-block dim-64 model
over thousands of windows of public-domain text on-device (avg loss 5.6 -> 4.7).

The embedding is a *gather* (~nlga-embed~): the per-window input is just seq
token + seq target indices (the ~embed-gather~ / ~embed-bwd~ / ~ce-grad-idx~
kernels index ~wte~ directly via a float->uint cast), not a seq x vocab one-hot.
~test/gpu-gather-test.el~ confirms it matches CPU autograd
(~photon-autograd-embedding~) to f32, and it cuts the corpus per-window time from
~38 ms to ~20 ms (vocab 336).  ~wte~ is still trained on-device by the
scatter-add backward.

* Layout

- ~lisp/nl-llm-arch.el~     -- core primitives (RMSNorm, RoPE, SiLU, SwiGLU)
- ~lisp/nl-llm-attn.el~     -- RoPE causal MHA/GQA + KV-cache decode
- ~lisp/nl-llm-moe.el~      -- top-k sparse mixture-of-experts FFN
- ~lisp/nl-llm-block.el~    -- pre-norm block + model forward (GQA/MoE)
- ~lisp/nl-llm-autograd.el~ -- reverse-mode backward for all of the above
- ~lisp/nl-llm-gpu.el~      -- GPU backend wiring (enable / invalidate)
- ~lisp/nl-llm-gpu-train.el~ -- on-device (resident) training step + SGD (MLP)
- ~lisp/nl-llm-gpu-ag.el~    -- resident-tensor autograd builder (full block on-device)
- ~test/*-test.el~          -- correctness + gradient checks (incl. gpu, gpu-train)
- ~examples/train-modern.el~      -- train a single-head RMSNorm/RoPE/SwiGLU block
- ~examples/train-modern-full.el~ -- train the full block (multi-head GQA + MoE)
- ~examples/bench-gpu.el~         -- CPU vs GPU forward/training timing table
- ~examples/bench-gpu-train.el~   -- on-device vs host-driven training timing
- ~examples/train-stacked-gpu.el~ -- train a multi-block model on-device
- ~examples/train-corpus-gpu.el~  -- on-device training over a whole corpus

* Run

#+begin_src sh
make test                 # all correctness + gradient-check suites (incl. GPU)
make train-modern-full    # train the full GQA + MoE block on open data
make bench-gpu            # CPU vs GPU timing table (needs a Vulkan device)
make bench-gpu-train      # on-device vs host-driven training timing
#+end_src

or directly:

#+begin_src sh
emacs -Q --batch -L lisp -L ../nelisp-photon/lisp -l test/autograd-test.el
#+end_src
